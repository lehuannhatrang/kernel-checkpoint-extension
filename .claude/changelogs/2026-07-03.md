# Changelog 2026-07-03

## Fix: restore reconnect broken after dropping KERNEL_ID

Root cause: a CRIU-restored kernel re-announces itself to Jupyter Enterprise
Gateway under the id it was checkpointed with (frozen in launcher memory). The
previous change commented out `KERNEL_ID` in the restore POST, so JEG minted a
new id while the restored pod reported the old one, and the two never matched.

### Changes

- `src/index.ts`
  - Re-enabled `KERNEL_ID = cpKernelId` in the restore `/api/kernels` POST
    (mandatory for the gateway reconnect to land).
  - Added kernel-id collision reconciliation before creating the restored
    kernel: query running kernels and act via `planRestore`
    (create / replace / refuse).
    - `replace` (checkpoint id is the notebook's own live kernel): confirm
      dialog, then shut it down and wait before recreating under the same id.
    - `refuse` (checkpoint id held by a different/other-notebook kernel):
      abort with a clear error; concurrent restore of one checkpoint is
      unsupported by the current launcher.
  - Added `shutdownKernelAndWait` helper.
  - Allowed opening the panel with no running kernel (removed the hard
    "No Kernel" block); restore now uses the checkpoint's stored `kernelName`
    (falls back to the spec name) instead of the current kernel's.
- `src/restore-collision.ts` (new) — pure `planRestore` decision helper.
- `src/__tests__/restore-collision.spec.ts` (new) — unit tests for the helper.
- `src/checkpoint-panel.tsx`
  - Pass the checkpoint's `kernelName` to `onRestore`.
  - Made `sessionId` optional on restore (older checkpoints lack it; it is not
    forwarded to the gateway today).
  - Disabled "Save Kernel State" when no kernel is running (create requires a
    live kernel; browsing/restoring does not).
- `src/types.ts` — added `kernelName` to the `onRestore` signature.

## Fix: panel would not open until the kernel finished loading

`execute` awaited `sessionContext.ready`, which blocks until kernel startup
completes, so the dialog only appeared after the kernel loaded. Removed the
await; the panel now opens immediately (kernel-derived values stay optional,
create is gated on `kernelId`). `src/index.ts`.

## Note: deployed 0.1.11 was built from stale (commented-out) source

The 0.1.11 wheel/image shipped a labextension bundle compiled while `KERNEL_ID`
was still commented out, so the restore POST carried no `KERNEL_ID` at runtime
even though it was later re-enabled in `.ts`. Editing TypeScript is not enough:
the webpack bundle under `kernel_checkpoint/labextension/static/` must be
rebuilt (`jlpm build:prod`) and repackaged. Rebuilt here and verified the
bundle now contains `KERNEL_ID`, `KERNEL_SESSION_ID`, `listRunning`,
`shutdownKernel`. A new release (bump version + Dockerfile pin, rebuild wheel +
image) is required to deploy the fix.

## Fix: busy-cell output not rendered after restore (hook timing)

Symptom: restoring a kernel that was mid-execution — the websocket receives the
resumed cell's messages but the notebook never renders them into the cell.

Root cause (hypothesis, evidence-backed via `@jupyterlab/services`
`KernelConnection._handleMessage`): `_iopubMessage.emit` fires for every iopub
message regardless of a matching future, so routing is not the issue. The
Jupyter server buffers a kernel's iopub while no client websocket is attached
and replays the backlog the instant a socket opens. The old code attached the
`RestoreHandler` iopub listener only *after* `await changeKernel()` resolved —
i.e. after the socket had already opened and flushed the backlog — so the
in-flight cell's output was missed.

Fix:
- `RestoreHandler` reworked to `arm(sessionContext, panel, busyCells)` which
  connects to `kernelChanged` and hooks the new kernel's iopub the moment the
  KernelConnection is created (before its websocket opens). `index.ts` arms the
  handler before `changeKernel` and falls back to a direct `hookKernel` if
  `kernelChanged` did not fire. `isHooked` guards the fallback.
- Added verbose diagnostics (`RESTORE_DEBUG`) logging arm/hook, and per-iopub
  match / no-match with parent ids, to confirm the fix or reveal a msg_id or
  cell-id mismatch on the cluster.

Not addressed here (likely infra / inherent, pending log evidence):
- Kernel shows busy and new cells queue until the resumed cell finishes — the
  restored kernel genuinely resumes the in-flight execution; the shell channel
  is FIFO-busy until it completes. Inherent to resuming mid-execution.
- Interrupt not working after restore — interrupt goes through EG's process
  proxy (SIGINT); after a CRIU restore the pid/pod mapping or signal handling
  may be stale. Extension-external (launcher / operator).

Verification: `tsc` passes; `jlpm build:prod` rebuilds the bundle
(kernelChanged/iopubMessage/connectionStatus present). End-to-end behavior
requires the live cluster; use the console diagnostics to confirm.
